# PTO ISA Manual Rewrite Plan

This document is the execution backlog for upgrading the PTO manual from a template-driven contract dump into a readable architectural manual that still holds up for implementers.

## Rewrite Goals

- make the manual explain why PTO is tile-first, not merely define that it is
- keep normative language only where a verifier, backend, or test can check it
- add at least one worked example to every rewritten chapter
- separate architecture-defined behavior from implementation-defined behavior explicitly
- keep English and Chinese chapters aligned in structure, examples, and terminology

## Wave Plan

### Wave 1

Canonical files:

- `docs/isa/README.md`
- `docs/isa/introduction/*.md`
- `docs/isa/machine-model/*.md`
- `docs/isa/programming-model/*.md`
- Chinese mirrors for the same files

Goals:

- replace the old preface template with a real introduction
- explain why PTO exists and why Auto and Manual both matter
- introduce a running example that the rest of the manual can reuse
- turn the programming chapter into a real guide instead of a checklist

### Wave 2

Canonical files:

- `docs/isa/state-and-types/*.md`

- `docs/isa/memory-model/*.md`
- Chinese mirrors for the same files

Goals:

- explain how legality is decided in practice
- make valid-region behavior concrete
- connect synchronization rules to real producer and consumer patterns

### Wave 3

Canonical files:

- `docs/isa/syntax-and-operands/*.md`
- `docs/isa/instruction-families/*.md`
- `docs/isa/tile/**`, `docs/isa/vector/**`, `docs/isa/scalar/**`, `docs/isa/comm/**`, and `docs/isa/system/**`
- `docs/isa/reference/*.md`
- Chinese mirrors for the same files

Goals:

- preserve toolchain precision while adding explanation-first chapter openings
- make backend profile and conformance language less mechanical
- keep verifier and interchange contracts precise without sounding autogenerated

## Chapter Checklist

Every rewritten chapter must include:

- one paragraph that says what problem the chapter answers
- one paragraph that explains why PTO chose this design over a simpler alternative
- one concrete example, preferably from `demos/`, `tests/`, or `docs/coding/tutorials/`
- one section that states the real normative contract
- one section that names common mistakes or non-portable assumptions

## Content To Delete Or Demote

When rewriting, remove or demote:

- sections that are only "scope / audience / conventions" without new information
- numbered subheadings that exist only to mirror a template
- bullet lists that replace explanation rather than support it
- `MUST/SHOULD/MAY` statements that are really style advice or philosophy
